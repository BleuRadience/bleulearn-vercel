# BleuLearn Platform — Developer Setup
AvaBleu House HQ | 2026

This file is the exact sequence of steps to take this codebase from "deployed static files" to "fully working with database, auth, and AI generation." Every step below was written by the same person who wrote the code, after actually running and testing it — not assumed from reading it.

## What works with ZERO setup
The entire static site — every strand page, every worksheet, every poster, every answer key, every lesson plan example — works the moment this is deployed to Vercel. No environment variables, no database, nothing to configure. This was true before this setup pass and remains true now.

## What requires setup, in order

### 1. Database (required for: login, signup, points, attendance, benchmarks, saved lesson plans)
1. In the Vercel dashboard, open this project, go to **Storage** → **Create Database** → **Postgres**.
2. Connect it to this project. Vercel automatically sets `POSTGRES_URL` and related variables — you do not type a connection string in by hand.
3. Set `MIGRATE_SECRET` in **Settings → Environment Variables** to any string you choose.
4. Redeploy (or it will pick up the new env vars on the next deploy automatically).
5. Run the migration once:
   ```
   curl -X POST https://YOUR-DOMAIN.vercel.app/api/migrate \
     -H "Authorization: Bearer YOUR_MIGRATE_SECRET"
   ```
   You should get back `{"success": true, "statementsRun": 11, ...}`. If you get a 503, `POSTGRES_URL` isn't connected yet — go back to step 2. If you get a 401, your `MIGRATE_SECRET` doesn't match what you set in step 3.
6. This is safe to run more than once. Every statement in `lib/schema.sql` is `CREATE TABLE IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS`.

### 2. Auth (required for: anyone actually logging in)
1. Generate a secret: `node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"`
2. Set it as `JWT_SECRET` in Vercel's environment variables.
3. That's it. Visit `/login/` and create the first account (recommend creating an `admin` account first).

### 3. AI Lesson Plan Generator (required for: /lesson-plans/ai-generator.html)
1. Get a key at console.anthropic.com/settings/keys.
2. Set it as `ANTHROPIC_API_KEY` in Vercel's environment variables.
3. Apply all three env vars above to Production, Preview, AND Development environments in Vercel — easy to set only one and wonder why preview deploys don't work.

## How to verify each piece actually works after setup
- `/api/auth/me` (GET, in browser) should return `{"user": null}` before logging in, and your user object after.
- Create one `teacher` account and one `scholar` account at `/login/`.
- Log in as the teacher. Use a tool like `curl` or Postman to `POST /api/points` with `{"scholar_id": 2, "points": 10, "reason": "test"}` (cookie from your teacher login must be sent).
- `GET /api/points?scholar_id=2` should now show that 10-point award in the ledger.
- Generate a lesson plan at `/lesson-plans/ai-generator.html` while logged in as that teacher. The response includes `"saved": true` if it was persisted. Check `GET /api/lesson-plans` to see it listed.

## What is genuinely NOT built, so you don't go looking for it
- **No password reset / forgot-password flow.** If someone forgets their password, an admin currently has to handle it directly in the database (`UPDATE users SET password_hash = ... WHERE email = ...` using a hash generated by `lib/auth.js`'s `hashPassword`). A real reset-via-email flow is future work.
- **No UI yet for teachers to browse/manage scholar rosters, award points through a button (only via direct API call), or view attendance/benchmark dashboards.** The API endpoints are real and tested; the front-end pages that call them for day-to-day teacher use are not yet built. This is the next logical phase of work, not done here.
- **No rate limiting on the AI generator.** Before any public launch, add basic per-user or per-IP request limits so one account can't run up an unbounded API bill.
- **No automated test suite.** Every endpoint here was manually tested during development (see the project's audit history for the exact verification steps used — including an in-memory Postgres run of the real schema and real queries). That is not a substitute for a real test suite if this grows further.

## Architecture decisions worth knowing before you change anything
- Shared helper code lives in `/lib`, never in `/api`. Anything placed directly in `/api` (including subfolders like `/api/auth/`) becomes a public endpoint on Vercel. This is why `db.js` and `auth.js` are in `/lib`.
- Auth is stateless JWT in an httpOnly cookie, not a sessions table. Tokens last 7 days. There is no server-side "log out everywhere" capability with this design — if you need that, you'd need to add a sessions table and check it on every request, which is a real architectural change, not a small patch.
- `points_ledger`, `attendance` (mostly), and `benchmark_scores` are designed as append/upsert tables matching the curriculum's own documented philosophy ("points are never subtracted, only added"). Don't add a DELETE or arbitrary UPDATE endpoint for points without re-reading why that constraint exists in the curriculum documentation first.

### What is honestly still not here, compared to real Moodle
This now covers Moodle's core teaching loop. It does not cover, and these are not small additions:
- **File upload submissions.** Submissions are text-only. Real file uploads need Vercel Blob storage provisioned (a separate step, like Postgres was) plus virus-scanning/size-limit handling — not built.

- **Full backup-and-restore of a course.** Not built.

- **A calendar UI.** Due dates exist as queryable data (`assignments.due_at`, `quizzes.available_until`) but there's no calendar page rendering them yet.
- **Badges and competency framework UI.** The curriculum's House System and Sovereign Surge cover some of this conceptually; there's no Moodle-style badge-issuing engine.
- **Manual override grading for short-answer quiz questions.** Auto-grading is exact-match only; a teacher cannot yet override a quiz auto-grade through a dedicated endpoint (assignments can be regraded via PATCH /api/submissions; quizzes currently cannot).

### New setup step: creating your first admin account
1. Set `INITIAL_ADMIN_SETUP_SECRET` in Vercel's environment variables to a random string.

## Tested Scenarios: Transfer Student and IEP Scholar

Asked to test these two specific real-world cases against the live schema and API logic. Both surfaced genuine gaps that have since been fixed.

### Transfer student
- **Found:** the chronic-absence calculation in `attendance.js` compared a scholar's absence count against a flat `18` (assuming a full 180-day year for everyone). A transfer student enrolled only ~25 days with 5 absences -- a real 20% absence rate, double the 10% policy threshold -- was never flagged, because `5 < 18`.
  **Fixed:** the threshold is now computed as 10% of the scholar's actual enrolled days (from their earliest `enrolled_at`), not a fixed number.
- **Found:** `enrollments` had no concept of a student leaving. A transferred-out scholar would remain on the active roster and in the gradebook class matrix forever, with no way to mark them gone without deleting (and losing) their grade and attendance history.
  **Fixed:** added `status` (`active`/`withdrawn`/`transferred_out`) and `withdrawn_at` to `enrollments`. Withdrawal is a `PATCH /api/enrollments`, not a delete -- history is preserved. Active rosters and the gradebook matrix now filter to `status = 'active'` by default; `?include_inactive=1` shows everyone. Re-enrolling a previously withdrawn scholar (a transfer-back) automatically restores active status.

### IEP / 504 scholar
- **Found:** there was no representation of an accommodation plan anywhere in the actual running system, despite the Master Curriculum document describing a full "Formal Accommodation Policy" in prose. A teacher had no way to even know a scholar had an IEP, let alone apply it.
- **Found, while building the fix:** quiz `time_limit_minutes` was stored on every quiz and enforced for **no one** -- a scholar submitting 3 hours after starting was accepted identically to one who submitted in 2 minutes. There was nothing to extend, because nothing was being timed.
  **Fixed both together:** new `accommodation_plans` table (plan type, extended time multiplier, status, review date). Quiz attempts now actually enforce `time_limit_minutes` for every scholar, and a scholar with an active plan gets that limit multiplied automatically at the moment they start the attempt. A late submission is rejected but **not** auto-graded as zero and **not** marked submitted -- it stays `in_progress` so a teacher can review it, since there's no manual quiz-grade-override endpoint yet (see below).

### Still open after this pass
- **No manual override for a quiz attempt that times out or needs a regrade.** Assignments can be regraded via `PATCH /api/submissions`; quizzes currently cannot. Named here rather than silently left for the next person to discover.
- **Accommodation plan management is restricted to teacher/admin**, with no dedicated special-education case-manager role. This platform's four roles (scholar/teacher/family/admin) don't distinguish that, and adding a fifth role for this alone was judged out of scope for this pass.
- **No notification is sent to a scholar's other teachers when an accommodation plan is created.** A scholar with five courses and one IEP plan is covered automatically in all five (the query checks `scholar_user_id`, not per-course), but no teacher is alerted that a new plan now applies to their class.

## Intake and Differentiation Assessments: Transfer, IEP, and Gifted

Three new real assessment workflows, each with database tables, tested API endpoints, and admin-panel UI. None of this existed before -- the prior `accommodation_plans` table only captured a time multiplier, which is an operational accommodation, not an assessment of how to teach anyone.

### New tables
`transfer_records`, `curriculum_gap_findings`, `placement_assessments`, `iep_evaluations`, `gifted_calibration_checkins`, `track_assignments`.

### New endpoints
| Endpoint | Purpose |
|---|---|
| `/api/transfer-records` | Intake record for a transferring scholar: prior school, prior curriculum notes |
| `/api/curriculum-gap-findings` | Per-strand documented gap between the prior school's coverage and BleuLearn's scope and sequence -- the direct answer to "review the other school's curriculum to see what they lack" |
| `/api/placement-assessments` | The actual administered diagnostic -- shared structure for a transfer diagnostic, a Gifted identification assessment, or a general placement check. **Automatically updates `track_assignments`.** |
| `/api/iep-evaluations` | Present level of performance, documented strengths/needs, and required recommended instructional strategies, tied to an accommodation plan -- this is what tells a teacher *how* to teach the scholar, not just how much extra time to give them |
| `/api/gifted-calibration` | Quarterly check-in: is the challenge level too easy, appropriate, or too challenging. **Automatically extends or rolls back the gifted track** based on the documented evidence -- this is the literal mechanism for "challenging enough to keep their attention but not overwhelming them," tested in both directions |
| `/api/track-assignments` | Read-only lookup of a scholar's current, evidence-backed track per strand. Deliberately has no edit endpoint -- the only way to change a track is to record a new assessment, so every track change always has documented evidence behind it |

### Real integration: the AI lesson generator now uses this
`POST /api/generate-lesson-plan` accepts an optional `scholar_user_id`. If provided, it looks up that scholar's current track for the relevant strand and tells the model to write differentiation that's actionable for a scholar *already confirmed* at that track, instead of generic Bridge/Grade-Level/Gifted boilerplate. Verified in isolation against a seeded `track_assignments` row before being wired into the live endpoint.

### How all three were verified
Full realistic scenarios run against the in-memory Postgres engine, using properly parameterized queries (an earlier draft of the test script itself had a raw-string quoting bug -- caught and fixed before trusting the result, the same discipline as the rest of this project):
- **Transfer:** intake record created, a documented gap recorded ("never seen a bar model, guesses operation from keywords"), a diagnostic assessment scoring Beginning (1), and confirmed the scholar's operative track became `bridge` -- matching both the gap finding and the diagnostic independently.
- **IEP:** an accommodation plan plus a full instructional evaluation with present level of performance, strengths, needs, and concrete recommended strategies (chunking, pre-filled WHO/WHEN, oral response option) -- a real teaching plan, not a multiplier.
- **Gifted:** identification assessment set the track to `gifted`, then a documented `too_challenging` calibration check-in correctly rolled the track *back* to `grade_level` -- confirming the system pulls back as readily as it escalates, and that both track-assignment rows survive in history rather than overwriting each other.

### Still open after this pass
- Accommodation-plan and IEP-evaluation creation remains teacher/admin only; there's no dedicated case-manager role, same limitation noted in the prior pass.
- No automated reminder when a `review_date` on an accommodation plan or a quarterly Gifted calibration check-in is due -- these are tracked as data but nothing currently surfaces "this is overdue" to a teacher or admin.
- `curriculum_gap_findings`' `bleulearn_expected_level` field is free text, not a structured link into the actual scope-and-sequence documents (BleuLearn_Math_Assessment_Framework.docx, etc.) -- a teacher filling this in is relying on their own knowledge of those documents, not a system-enforced cross-reference.

## AI-Assisted, Bias-Aware Review with Mandatory Human Oversight

Every placement assessment, Gifted calibration check-in, IEP evaluation, and curriculum gap finding now generates an independent AI read of the documented evidence (`lib/aiReview.js`) before anything becomes operative. **The AI never decides. It writes an `ai_assessment_reviews` row with `status='pending_review'`. A teacher or admin must explicitly approve or override via `/api/admin/review-pending` before a scholar's track, accommodation, or placement actually changes.**

### What the AI is given, and what it is deliberately never given
This platform collects no race, ethnicity, gender, or other demographic field anywhere in its schema -- so the AI cannot be biased by data it was never given. It receives only: the documented evidence text a human assessor wrote, the recommended action, and (where relevant) the structured scope-and-sequence expectation. The system prompt explicitly instructs it to flag vague or impressionistic evidence ("seems behind," "is clearly gifted" with no specific observed instance) rather than accept it -- because that pattern is the documented mechanism behind real-world disparities in gifted and IEP referral, not a guess about who a student is.

**Honest limit, stated plainly:** this is a second, independent, evidence-anchored opinion -- not a debiasing guarantee. An AI model is not immune to bias, and this system makes no claim that it is. What it does claim, and what was tested, is that it never has access to anything that would let it discriminate by identity, and that a human always has the final word.

### New tables and endpoints
- `ai_assessment_reviews` -- the pending/approved/overridden record. Overriding requires `human_decision_notes` -- there is no silent-click path.
- `scope_sequence_references` -- structured, documented scope-and-sequence entries a curriculum gap finding can cite, instead of relying on one teacher's memory of the framework documents (closes a previously-named gap).
- `/api/admin/review-pending` (GET lists pending items with full evidence joined in; POST approves or overrides)
- `/api/admin/equity-analytics` -- referrer-level Gifted/Bridge referral counts, accommodation-plan creation counts by creator, and AI-disagreement/override rates. No demographic breakdown is possible or attempted; this surfaces a question, not a verdict.
- `/api/admin/overdue-reviews` -- overdue accommodation plan reviews and scholars on Gifted track missing this quarter's calibration check-in (closes a previously-named gap).
- New role: `case_manager`, with the same access as teacher/admin to all assessment endpoints, onboarded through the same invite-code system as teacher/admin (closes a previously-named gap).

### Two real bugs found while testing this, fixed before shipping
1. **`ai_agrees_with_human` was declared `NOT NULL`**, but the AI-unavailable fallback path (the actual current state of this platform -- no `ANTHROPIC_API_KEY` is configured yet) legitimately needs to insert `NULL` for "no opinion given," a real third state distinct from agree/disagree. Every assessment submission would have crashed until this was caught by running the actual insert, not just reading the code.
2. **The `users.role` and `invite_codes.role` CHECK constraints couldn't be reliably updated by guessing the existing constraint's name.** Testing this against an in-memory Postgres engine proved the guessed name (`users_role_check`, the documented Postgres default) did not match what that engine actually used internally -- meaning a name guess cannot be trusted blindly even when it follows documented convention, and there is no fully reliable way to discover the real name through that same engine's catalog views either. The actual fix: the original `CREATE TABLE` statements were updated to include `case_manager` from the start (this platform has not been provisioned in production yet, so this covers the realistic case completely), with a best-effort, explicitly-labeled `ALTER` fallback and a manual verification command documented for the one scenario it can't guarantee.
3. **A third issue, isolated rather than assumed:** `COUNT(*) FILTER (WHERE ...)` against a `NULL`-valued boolean column returned an incorrect count in the in-memory test engine, while a plain `WHERE` clause with the identical condition was correct. Rather than assume real Postgres has the same bug (likely it does not -- `FILTER` is a long-standing, well-supported feature) or assume the test engine is simply wrong (unverifiable without a real Postgres instance), the equity-analytics queries were rewritten to use `SUM(CASE WHEN ... THEN 1 ELSE 0 END)`, which was verified correct in both the plain-`WHERE` form and the rewritten form -- removing the ambiguity entirely instead of betting on which engine to trust.

### Still open after this pass
- The AI review's bias-risk-flagging is a heuristic based on the model's read of evidence quality, not a validated bias-detection instrument -- it should not be cited as one in any compliance or legal context.
- Equity analytics shows referrer-level counts but no statistical significance testing or peer-comparison baseline -- an admin still has to apply judgment to what counts as an outlier.
- There is no UI yet for browsing `scope_sequence_references` or adding new entries to it -- it currently has to be populated via direct database access or a future endpoint.

## Platform Migration: Off Vercel, Onto Any Node.js Host

This platform no longer depends on Vercel. It was rewritten to run as a plain Express server, which works identically on Render, Railway, Coolify, a bare VPS, or your own laptop. Nothing about the 37 API handler files changed -- only `lib/db.js` (now uses the standard `pg` package instead of `@vercel/postgres`) and the new `server.js` (a plain Express router that auto-discovers every file in `/api` and mounts it, replacing Vercel's filesystem-based routing).

### Why this migration happened
Vercel's serverless model runs backend code as short-lived functions with cold starts and execution-time limits -- it has no first-class way to run a persistent backend service. For a full-stack app with a real database and long-lived sessions, that's a structural mismatch, not a configuration problem. Render, Railway, and Coolify all run actual Node.js processes instead, which is what this app was always architecturally closer to.

### Recommended: Render
Render was chosen as the primary recommendation because: it's a real Node.js runtime (no rewrite needed beyond what's already done), it has managed Postgres on the same platform, and its pricing is flat and predictable -- Starter web service $7/month, Starter Postgres $7/month, no usage-based billing that can spike unexpectedly the way Vercel's can.

**Deploy steps:**
1. Push this repository to GitHub.
2. In Render: New -> Blueprint -> connect the repo. `render.yaml` (included) auto-configures the web service and database.
3. Render sets `DATABASE_URL` automatically from the attached database -- you do not type a connection string by hand.
4. Set `JWT_SECRET` (render.yaml auto-generates one) and `ANTHROPIC_API_KEY` (you must add this yourself -- get it at console.anthropic.com) in the Render dashboard's Environment tab.
5. Once deployed, run the one-time migration the same way as before: `curl -X POST https://your-app.onrender.com/api/migrate -H "Authorization: Bearer YOUR_MIGRATE_SECRET"`.

### Alternative: Coolify on a $5/month VPS -- full infrastructure ownership
If owning your own infrastructure matters more than convenience -- which is a real, legitimate fit for a curriculum whose entire stated philosophy is sovereignty rather than dependency on someone else's platform -- Coolify is free, open-source software (55,900+ GitHub stars) that you run on a basic VPS (DigitalOcean, Hetzner, or Linode, roughly $5/month). It automates SSL, Git-based deploys, and monitoring, so you get Vercel-style convenience without renting space on someone else's platform. The tradeoff: you are responsible for the underlying server's security updates, which Render handles for you. Deployment steps are the same `npm install && npm start` Coolify auto-detects from this repo's `package.json` -- no platform-specific config file needed.

### What did NOT need to change
Every one of the 37 API handler files -- authentication, courses, gradebook, the AI-assisted bias review workflow, the transfer/IEP/Gifted assessment system, all of it -- was already written using `req.method`, `req.query`, `req.body`, and `res.status().json()`, which is the same shape both Vercel's serverless functions and a plain Express server use. That design choice, made early in this project for unrelated reasons, turned out to be exactly what made this migration low-risk: only the database connection layer and the routing layer were platform-specific. The business logic never was.

### How this migration was verified
- The new SQL query-builder in `lib/db.js` was tested against a real Postgres-compatible engine, including a SQL injection attempt, to confirm it produces safe, correctly parameterized queries identical in behavior to the code it replaced.
- The actual `server.js` was started and hit with real HTTP requests: the health check, static file serving, a 404 case, and a JSON POST body all confirmed working.
- A full real-world flow was run end-to-end through the live server: admin bootstrap signup, receiving a real session cookie, using that cookie to create a course (an authenticated database write), and reading it back out -- the same privilege-escalation protection from the original security audit was confirmed still active in this exact flow (a teacher signup without an invite code was correctly rejected with 403 before the legitimate admin flow was tested).
- One real process mistake was caught and corrected during this testing: an `npm install` run without explicitly listing a test-only dependency pruned it from `node_modules`, breaking a subsequent test run. This was diagnosed and fixed before relying on the result, rather than the test result being silently trusted.

## Coolify Deployment -- Complete Step-by-Step Guide (Decided On)

Coolify v4.0.0 left beta and shipped stable on April 27, 2026, after roughly two years in beta -- this is the current, mature version these steps describe, not a beta workflow that might change underneath you.

### Step 1: Get a VPS (~$5/month)
Any provider works -- Hetzner, DigitalOcean, Linode/Akamai, or RamNode all offer a basic VPS in the $4-6/month range with hourly billing available on some, so you can test before committing to a full month. Minimum spec for this app: 1 CPU, 2GB RAM. This platform is a plain Express server plus static files -- it is far lighter than a typical Next.js app, so you do not need the larger VPS sizes sometimes recommended for heavier frameworks.

### Step 2: Install Coolify (one command, ~5 minutes)
SSH into your fresh VPS and run the single install command from `coolify.io/docs/get-started/installation`. It installs Docker, Traefik (the reverse proxy), and the Coolify dashboard automatically. When it finishes, the dashboard is reachable at `http://YOUR_SERVER_IP:8000`.

**Before going further:** open your VPS provider's firewall settings and allow port 8000 (the Coolify dashboard) and port 443/80 (for your deployed app's HTTPS traffic once you add a domain).

### Step 3: Add the PostgreSQL database
In the Coolify dashboard: create a Project, then **Add New Resource -> Database -> PostgreSQL**. Choose PostgreSQL 16 (the current stable default). Click Start. Coolify provisions it in its own container and generates credentials automatically -- copy the **internal connection string** shown in that resource's "Connection Information" panel. It will look like `postgresql://postgres:PASSWORD@postgresql-xxxxx:5432/postgres`. Use the *internal* hostname, not a public IP -- your app and the database share Coolify's private Docker network, so internal traffic never touches the public internet.

### Step 4: Add this application
**Add New Resource -> Application -> Git Repository** (GitHub, GitLab, or any Git source). Point it at this repository. Coolify auto-detects this is a plain Node.js app via Nixpacks -- or, since a `Dockerfile` is included in this repository, Coolify will use that automatically instead for a more deterministic build. Either path works; the Dockerfile is provided specifically to remove any guessing.

Set these fields in the application's settings:
- **Port Exposes:** `3000` (matches what `server.js` listens on and what the Dockerfile exposes)
- **Health Check Path:** `/api/hello` (a real, public, no-auth endpoint already in this codebase -- built for exactly this purpose)

### Step 5: Set environment variables
In the application's Environment Variables tab, add:
```
DATABASE_URL = <the internal connection string from Step 3>
JWT_SECRET = <generate with: node -e "console.log(require('crypto').randomBytes(48).toString('hex'))">
ANTHROPIC_API_KEY = <from console.anthropic.com>
INITIAL_ADMIN_SETUP_SECRET = <any random string, for your one-time first-admin bootstrap>
NODE_ENV = production
```

### Step 6: Deploy, then run the one-time migration
Click Deploy. Watch the build logs -- you should see the same "Mounted /api/..." lines for all 37 routes that were confirmed locally during testing. Once it's live:
```
curl -X POST https://your-coolify-domain/api/migrate \
  -H "Authorization: Bearer YOUR_MIGRATE_SECRET"
```
(Set `MIGRATE_SECRET` as another environment variable first, same as documented earlier in this file.)

### Step 7: Custom domain and HTTPS
Coolify's built-in Traefik proxy handles Let's Encrypt SSL automatically once you point your domain's DNS at your server and add it in the application's Domains tab. No separate certificate management needed.

### Two specific failure modes checked against this exact codebase before writing this guide
Coolify's own documentation calls out two things that silently break a deployment that otherwise looks fine, and both were checked directly against this app rather than assumed safe:
1. **The app must listen on `0.0.0.0`, not `127.0.0.1`** -- Traefik connects over the internal Docker network, not localhost, so a server only accepting local connections is unreachable even though the container reports as running. `server.js` previously relied on Node's unstated default (which is `0.0.0.0`, but unstated). It now binds explicitly, and this was re-tested afterward: the server was started fresh and connected to successfully on both `localhost` and `127.0.0.1`.
2. **The health check needs a real endpoint, or it should be disabled** -- a failing health check makes Traefik stop routing to the container even if the app is otherwise fine. `/api/hello` already existed in this codebase from the Vercel-era health check and requires no auth, making it a direct fit -- it is now documented as the explicit health check path above rather than left for a developer to discover the need for one mid-deployment.

### How the Docker build path was verified without Docker itself
Docker is not available in the environment this was developed in, so the Dockerfile's logic was verified differently: every dependency it would install (`npm install --omit=dev`) was confirmed to match exactly what's declared in `package.json` (no missing runtime dependency hiding in `devDependencies`), and then the *exact* install command the Dockerfile runs was executed for real against a freshly wiped `node_modules`, followed by actually starting the server and confirming it answers requests -- the same end state a Docker build would produce, verified through the parts of it that don't require Docker specifically to test.
